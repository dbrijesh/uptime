Private vs Public GitHub Runners

Feature Public Runners Private Runners
Cost Free for public repos, limited minutes for private You manage infrastructure costs
Security Shared environment, limited access Dedicated, secure environment
Performance Standard GitHub-managed specs Customizable hardware specs
Setup Zero setup required Manual installation and configuration
Network Access Internet access only Can access internal networks

When to Use Public Runners

  • Open source projects
  • Standard CI/CD needs
  • Quick prototyping
  • Limited budget

When to Use Private Runners

  • Enterprise security requirements
  • Access to internal resources
  • Custom hardware needs
  • Heavy computational workloads

Setup Proxy for GitHub Access

Corporate Network Considerations

Many corporate environments require proxy configuration for external access.

Runner Configuration

# Environment variables for proxy export HTTP_PROXY=http://proxy.company.com:8080 export HTTPS_PROXY=http://proxy.company.com:8080 export NO_PROXY=localhost,127.0.0.1,.company.com # Git configuration git config --global http.proxy http://proxy.company.com:8080 git config --global https.proxy http://proxy.company.com:8080

Docker Proxy Configuration

# ~/.docker/config.json { "proxies": { "default": { "httpProxy": "http://proxy.company.com:8080", "httpsProxy": "http://proxy.company.com:8080", "noProxy": "localhost,127.0.0.1,.company.com" } } }
Pro Tip: Always test proxy connectivity before deploying runners in production. Use tools like curl or wget to verify external access.

GitHub Actions & Workflows

What are GitHub Actions?

GitHub Actions is a CI/CD platform that allows you to automate your build, test, and deployment pipeline directly in your GitHub repository.

Typical Repository Structure

my-project/ ├── .github/ │ └── workflows/ │ ├── ci.yml │ ├── deploy.yml │ └── release.yml ├── src/ │ └── MyApp/ ├── tests/ │ └── MyApp.Tests/ ├── Dockerfile ├── README.md └── .gitignore

Workflow Components

Trigger
→
Job
→
Steps
→
Actions
  • Workflows: Configurable automated processes
  • Jobs: Set of steps that execute on the same runner
  • Steps: Individual tasks within a job
  • Actions: Reusable units of code

GitHub Secrets & PAT Tokens

What are GitHub Secrets?

Encrypted environment variables stored in your repository or organization settings. They're used to store sensitive information like API keys, passwords, and tokens.

Personal Access Tokens (PAT)

Alternative to using passwords for Git over HTTPS, providing fine-grained access control.

# Creating a PAT Token: # 1. GitHub Settings → Developer settings → Personal access tokens # 2. Generate new token (classic or fine-grained) # 3. Select appropriate scopes: # - repo (for private repositories) # - workflow (for GitHub Actions) # - packages (for GitHub Packages)

Using Secrets in Workflows

name: CI on: [push] jobs: build: runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 with: token: ${{ secrets.GITHUB_TOKEN }} - name: Login to Docker Hub uses: docker/login-action@v3 with: username: ${{ secrets.DOCKER_USERNAME }} password: ${{ secrets.DOCKER_PASSWORD }}
Security Best Practice: Never commit secrets to your repository. Always use GitHub Secrets or external secret management systems.

.NET Build with Docker Workflow

name: .NET Docker Build on: push: branches: [ main, develop ] pull_request: branches: [ main ] env: REGISTRY: ghcr.io IMAGE_NAME: ${{ github.repository }} jobs: build-and-test: runs-on: ubuntu-latest steps: - name: Checkout code uses: actions/checkout@v4 - name: Setup .NET uses: actions/setup-dotnet@v4 with: dotnet-version: '8.0.x' - name: Restore dependencies run: dotnet restore - name: Build run: dotnet build --no-restore --configuration Release - name: Test run: dotnet test --no-build --configuration Release --logger trx - name: Log in to Container Registry uses: docker/login-action@v3 with: registry: ${{ env.REGISTRY }} username: ${{ github.actor }} password: ${{ secrets.GITHUB_TOKEN }} - name: Build and push Docker image uses: docker/build-push-action@v5 with: context: . push: true tags: | ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

Running Actions: Automated vs Manual Triggers

Automated Triggers

# Push/Pull Request on: push: branches: [ main, develop ] pull_request: branches: [ main ] # Scheduled (Cron) on: schedule: - cron: '0 2 * * 0' # Every Sunday at 2 AM # Repository Events on: release: types: [published] issues: types: [opened, labeled]

Manual Triggers

# Workflow Dispatch (Manual) on: workflow_dispatch: inputs: environment: description: 'Environment to deploy' required: true default: 'staging' type: choice options: - staging - production logLevel: description: 'Log level' required: true default: 'warning' type: choice options: - info - warning - debug jobs: deploy: runs-on: ubuntu-latest steps: - name: Deploy to ${{ inputs.environment }} run: echo "Deploying to ${{ inputs.environment }} with log level ${{ inputs.logLevel }}"

Automated Benefits

  • Consistent execution
  • Immediate feedback
  • Reduced human error
  • Continuous integration

Manual Benefits

  • Controlled deployments
  • Parameter customization
  • Testing scenarios
  • Emergency procedures
