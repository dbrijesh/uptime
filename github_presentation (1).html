‚úÖWhen to Use Public Runners

  • Open Source Projects: Free and feature-rich for public repositories
  • Standard Development: Common languages and frameworks supported
  • Quick Prototyping: Zero setup time, immediate availability
  • Small Teams: Cost-effective for limited usage patterns
  • Simple CI/CD: Build, test, deploy to cloud services
  • Learning/Education: Perfect for understanding GitHub Actions
Best Practice: Start with public runners and migrate to private only when specific needs arise

üè¢When to Use Private Runners

  • Enterprise Security: Compliance requirements (SOC2, HIPAA, PCI)
  • Internal Resources: Database access, internal APIs, on-premises systems
  • High Performance: Machine learning, large builds, intensive testing
  • Custom Environment: Proprietary tools, specific OS versions
  • Cost Optimization: Heavy usage patterns (>10,000 minutes/month)
  • Air-Gapped Networks: Completely isolated environments
Consideration: Private runners require ongoing maintenance, security patching, and infrastructure management

Runner Configuration Examples

Public Runner Usage

jobs: build: runs-on: ubuntu-latest # or windows-latest, macos-latest steps: - uses: actions/checkout@v4 - name: Build application run: npm install && npm run build

Private Runner Usage

üí°Enterprise Strategy: Use public runners for open source and development environments, private runners for production deployments and sensitive workloads

Comprehensive Proxy Configuration

Environment Variables Setup

# System-wide proxy configuration export HTTP_PROXY=http://proxy.company.com:8080 export HTTPS_PROXY=http://proxy.company.com:8080 export NO_PROXY=localhost,127.0.0.1,*.company.com,*.internal,10.0.0.0/8 # Case-sensitive alternatives export http_proxy=$HTTP_PROXY export https_proxy=$HTTPS_PROXY export no_proxy=$NO_PROXY # Authenticated proxy (if required) export HTTP_PROXY=http://username:password@proxy.company.com:8080 export HTTPS_PROXY=http://username:password@proxy.company.com:8080

Git Proxy Configuration

# Global Git proxy settings git config --global http.proxy $HTTP_PROXY git config --global https.proxy $HTTPS_PROXY # SSL verification (corporate certificates) git config --global http.sslverify true git config --global http.sslcainfo /path/to/corporate-ca.crt # Per-domain proxy settings git config --global http.https://github.com.proxy $HTTPS_PROXY git config --global http.https://internal.company.com.proxy "" # Timeout configurations git config --global http.lowSpeedLimit 1000 git config --global http.lowSpeedTime 300

Docker Proxy Configuration

# ~/.docker/config.json { "proxies": { "default": { "httpProxy": "http://proxy.company.com:8080", "httpsProxy": "http://proxy.company.com:8080", "noProxy": "localhost,127.0.0.1,*.company.com,*.internal" } }, "experimental": true } # Dockerfile proxy arguments ARG HTTP_PROXY ARG HTTPS_PROXY ARG NO_PROXY # Docker daemon configuration (/etc/docker/daemon.json) { "proxies": { "http-proxy": "http://proxy.company.com:8080", "https-proxy": "http://proxy.company.com:8080", "no-proxy": "localhost,127.0.0.1,*.company.com" } }

Core Components Explained

üìãWorkflows

Definition: YAML files defining automated processes

  • Stored in .github/workflows/
  • Event-driven execution
  • Configurable triggers and conditions
  • Support for complex logic and dependencies

‚ö°Jobs

Definition: Groups of steps executed on the same runner

  • Run in parallel by default
  • Can specify dependencies
  • Fresh environment per job
  • Configurable runner requirements

üîÑSteps

Definition: Individual tasks within a job

  • Execute sequentially
  • Can run shell commands or actions
  • Share environment and filesystem
  • Support conditional execution

üß©Actions

Definition: Reusable units of code

  • JavaScript, Docker, or composite actions
  • Available from GitHub Marketplace
  • Version-controlled and parameterizable
  • Custom actions for specific needs

Enterprise Secret Management Strategy

Security in CI/CD pipelines extends beyond basic credential storage. Enterprise environments require comprehensive secret management that includes encryption, rotation, auditing, and compliance with regulatory requirements.

üîíSecret Types & Scope

  • Repository Secrets: Single repository access
  • Environment Secrets: Environment-specific credentials
  • Organization Secrets: Shared across repositories
  • Dependabot Secrets: Dependency update credentials

üîÑRotation & Lifecycle

  • Automated Rotation: Scheduled credential updates
  • Expiration Tracking: Monitor credential lifecycle
  • Emergency Revocation: Immediate secret invalidation
  • Audit Trails: Complete usage history

üìäCompliance & Governance

  • Access Control: Role-based secret permissions
  • Approval Workflows: Multi-stage secret deployment
  • Compliance Reporting: Regulatory requirement tracking
  • Integration: External secret management systems

üõ°Ô∏èSecurity Controls

  • Encryption at Rest: AES-256 encryption
  • Encryption in Transit: TLS 1.2+ protocols
  • Access Logging: Comprehensive audit logs
  • Principle of Least Privilege: Minimal access rights
